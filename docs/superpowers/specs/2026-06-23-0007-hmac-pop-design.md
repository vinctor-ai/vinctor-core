# ADR 0007 hardening C3 — HMAC proof-of-possession for subject tokens (Design)

Date: 2026-06-23
Status: Design (founder chose HMAC-PoP over asymmetric DPoP / defer, 2026-06-23).
Next: adversarial spec review → writing-plans → TDD.

## Goal

Close the residual replay gap for subject tokens (a token captured in transit is,
within its TTL, usable until expiry/revoke) with a **stdlib-only, symmetric
proof-of-possession**: each delegated enforce must carry a fresh, per-request HMAC
proof produced by the agent that holds a per-token secret. A party that steals the
raw token (but not the secret) cannot produce a valid proof, and a captured proof
cannot be replayed (freshness + nonce). No new dependency (`hmac`/`hashlib`/
`secrets` only).

## Threat model / what this buys

- The `pop_secret` is generated by Vinctor at mint and returned **once** to the
  agent (like the raw token). It is held by the **agent**, not the PEP.
- The agent computes a per-request proof and hands `{token, proof}` to the PEP; the
  PEP relays both. So a PEP (or anyone holding only the token) **cannot** act
  without a fresh agent-produced proof — sender-constraint in the delegated model.
- A network capture of `{token, proof}` cannot be replayed (nonce cache +
  freshness) and cannot be retargeted to a different `(action, resource)` (the
  proof binds them).

**Non-goals (deferred):** asymmetric DPoP (`cryptography`/PyJWT) where Vinctor
never holds the secret; an operator `require_pop` mandate flag (mirrors
`require_subject_token` — follow-up); a cross-process/durable replay cache
(this slice is per-process, single-node — see Risks); HTTP method/path binding
(the proof binds the *authorization* `(action, resource)`, not the transport).

## Design

### Opt-in per token (default-off)
PoP is requested by the agent at mint. A token with a stored `pop_secret` is
**pop-required**; a token without one behaves exactly as today. There is no global
or per-agent mandate in this slice.

### Schema v7
Add one nullable column `pop_secret TEXT` to `subject_tokens` (additive,
PRAGMA-guarded ALTER helper + CREATE TABLE body, v7 `schema_migrations` row).
Presence of `pop_secret` ⇒ pop-required. (Bump all non-empty-DB schema-version
asserts to include 7.)

**At-rest note:** `pop_secret` is stored in plaintext SQLite (HMAC verification
needs the secret, not a hash). Accepted for the local/preview posture and
mitigated by: it is per-token and short-lived, the *raw token* (never stored, only
its hash) is also required to use it, and tokens are revocable. A future hardening
can derive per-token secrets from a server master key. Documented limitation, not
a blocker.

### Mint
- `POST /v1/tokens` body gains optional `pop: true` (boolean). When true, mint
  generates `pop_secret = secrets.token_urlsafe(32)`, stores it on the token, and
  returns it **once** in the 201 body as `pop_secret` (alongside `token`).
- CLI `vinctor agent token mint --pop` requests it; the once-shown output includes
  the `pop_secret` with a "store now, unrecoverable" warning (same convention as
  the raw token).
- `pop` defaults false → no secret, not pop-required.

### Proof header
The relay sends `X-Subject-Token-Proof: <ts>.<nonce>.<mac>` where:
- `ts` = integer Unix seconds (the agent's clock at proof time).
- `nonce` = `secrets.token_urlsafe(12)` (per-proof, enables legitimate retries).
- `mac` = `urlsafe_b64encode(HMAC_SHA256(pop_secret, canonical)).rstrip("=")`,
  `canonical = "\n".join([action, resource, str(ts), nonce, token_id])`.

`token_id` (public, from mint) ties the proof to the token; `action`/`resource`
tie it to the operation; `ts`/`nonce` give freshness + anti-replay.

### Verification (delegated enforce, only when `token.pop_secret is not None`)
After the token is resolved and the identity tuple + (C2) binding pass, and only
for a pop-required token:
1. The proof header must be present (else fail closed).
2. Parse `<ts>.<nonce>.<mac>` (exactly 3 parts, ts an int) — malformed → fail closed.
3. Freshness: `abs(int(now_unix) - ts) <= POP_SKEW_SECONDS` (default **60**;
   `VINCTOR_SUBJECT_TOKEN_POP_SKEW_SECONDS`). `now` is the service clock passed to
   the contract.
4. Recompute `mac` over `(request.action, request.resource, ts, nonce,
   token.token_id)` with the stored `pop_secret`; compare with
   **`hmac.compare_digest`** (constant-time). Mismatch → fail closed.
5. **Replay:** a bounded in-memory cache keyed `(token_id, nonce)`; if already seen
   (within the freshness window) → fail closed; else record. The cache evicts
   entries older than the skew window and is size-capped (drop-oldest) so it cannot
   grow unboundedly under load. Per-process (see Risks).

**All failures** (missing / malformed / stale / bad-mac / replayed) return the
**same generic** `403 forbidden` + `subject_token_invalid` rejection audit as the
other token failures — no distinction leaked (consistent with revoked==invalid).
A pop-required token with no/blank proof is one of these failures.

### Default-off / ordering
The PoP check runs **inside** the existing optional-token block, after the
identity-tuple (and C2 bound) checks, and **only** when `token.pop_secret is not
None`. A non-pop token (`pop_secret` NULL) skips it entirely → behavior byte-for-
byte unchanged. The `require_subject_token` mandate (C1) is orthogonal and runs
earlier.

### Wiring
- `V1DelegatedEnforceRequest` gains `subject_token_proof: str | None = None`;
  `handle_v1_delegated_enforce_http` reads `x-subject-token-proof` and threads it.
- `delegated_enforce_v1_contract` does the verification (needs `now`, already
  passed). The replay cache is a small module-level/bounded structure owned by the
  service layer (mirroring `AuthFailureAuditThrottle`'s per-process model) and
  passed in or held on the service; keep it injectable for tests.
- Mint plumbs `pop` through the contract/service/HTTP/CLI like the C2 binding.

## Testing (TDD)
- **Mint:** `pop:true` → 201 returns `pop_secret` (urlsafe), token is pop-required;
  default (no `pop`) → no secret, not pop-required; CLI `--pop` once-shown output.
- **Verify (contract-level, compute a real HMAC in the test):**
  - valid proof (fresh ts, correct mac, matching action/resource) → permit,
    `identity_proven=true`.
  - pop-required token + **no** proof → fail closed `subject_token_invalid`.
  - bad mac (wrong secret) → fail closed.
  - stale ts (now-ts > skew) → fail closed; future ts beyond skew → fail closed.
  - proof for a **different** action/resource than the request → fail closed.
  - **replay**: same `(token_id, nonce)` presented twice → second fails closed;
    a retry with a **new** nonce (same op, fresh ts) → permits.
  - malformed proof (wrong part count / non-int ts) → fail closed.
  - **non-pop token** (no `pop_secret`) ignores any proof header and behaves as
    today (regression).
- **No-leak:** the failure reason/response never reveals pop-required vs bad-mac
  vs replay; `pop_secret` never appears in any response/audit except the one-time
  mint body.
- **Schema v7** recorded; asserts bumped.

## Risks
- **Per-process replay cache:** a multi-process/multi-node deployment would not
  share the nonce cache, so within the (short) skew window an exact replay could
  succeed on a different process. Acceptable for the single-node local/preview
  posture; a durable/shared cache is a follow-up. Document it.
- **Clock skew:** the agent and server clocks must agree within
  `POP_SKEW_SECONDS`; too tight breaks legitimate use, too loose widens replay.
  Default 60s, configurable.
- **At-rest secret:** see the schema note.

## Review-driven revisions (2026-06-23, adversarial spec review)

These refine/override the design above; the plan implements these.

1. **Injection-proof canonicalization + validate-before-MAC.** Do NOT
   `"\n".join(...)` (boundary-shift forgery via a field containing the delimiter).
   Use length-prefixed bytes: `canonical = b"".join(len(f.encode()).to_bytes(4,
   "big") + f.encode() for f in [action, resource, str(ts), nonce, token_id])`.
   The agent and verifier MUST use the identical construction. Additionally,
   **validate `request.action`/`request.resource` (`is_valid_requested_action`/
   `is_valid_requested_resource`) BEFORE computing the MAC** so the binding's
   integrity never depends on downstream validation. Test: a `\n` in `resource`
   must not enable cross-op forgery.
2. **Replay honesty (multi-process).** Anti-replay holds **single-process only**.
   Default skew **30s** (`DEFAULT_SUBJECT_TOKEN_POP_SKEW_SECONDS=30`), and the
   effective replay window is **2×skew**. State plainly: with >1 worker the
   per-process nonce cache does not share, so an exact replay can succeed on
   another process within 2×skew. A shared/durable nonce cache is a **named
   follow-up** (add to next-actions). Single-node local/preview is the supported
   posture.
3. **Replay cache correctness > memory.** Eviction is **time-based only**: purge
   entries whose `ts < now - skew` (they can never pass freshness again, so
   dropping them is safe). The size cap is a **back-pressure/DoS control**: if the
   cache is full of still-fresh entries, **fail closed** (reject the new proof) —
   never drop-oldest a fresh entry. **Record the `(token_id, nonce)` only AFTER the
   MAC + freshness checks pass** (a bad-mac/stale proof must not consume cache
   slots — otherwise it's a cheap eviction-DoS). Test: cache full of fresh entries
   → reject, not silently allow.
4. **Replay cache is an injectable instance**, mirroring
   `AuthFailureAuditThrottle`: a new `PopReplayCache` class constructed in each
   service's `__post_init__` (`self._pop_replay = PopReplayCache()`), passed as an
   explicit keyword param `pop_replay_cache: PopReplayCache | None = None` to
   `delegated_enforce_v1_contract`. **Not** a module-level singleton (tests
   construct their own per test).
5. **Clock:** `now_unix = int(now.timestamp())`; the contract requires a tz-aware
   `now` (normalize `if now.tzinfo is None: now = now.replace(tzinfo=UTC)` at the
   top of the PoP check). The agent's proof `ts` is integer Unix seconds.
6. **Config threading (no env in the contract).** Read
   `VINCTOR_SUBJECT_TOKEN_POP_SKEW_SECONDS` in `local_http` (mirror the `max_ttl`
   precedent), default from `service_config.DEFAULT_SUBJECT_TOKEN_POP_SKEW_SECONDS`,
   pass `pop_skew_seconds` into `handle_v1_delegated_enforce_http` and thread it as
   an explicit contract kwarg (default 30). The contract never reads `os.environ`.
7. **Mint plumb-through (full list):** add `pop` to the `_parse_tokens_body` allowed
   field set (else 400 "unexpected field") and validate it is a real `bool` (reject
   bool/int confusion, mirror the `ttl_seconds` guard), default False; add
   `pop_secret` to `SubjectTokenMintResult`; add `pop` to the `V1TokenService`
   Protocol + `handle_v1_tokens_http` call + both services' `mint_subject_token`;
   add `pop_secret` to the 201 body; CLI `--pop` (store_true) + the `pop_secret`
   line in the once-shown output.
8. **Proof header threading:** add `subject_token_proof: str | None = None` as the
   final field of `V1DelegatedEnforceRequest` (and update its stale docstring);
   read `x-subject-token-proof` in `handle_v1_delegated_enforce_http`; the contract
   treats `proof is None or proof.strip() == ""` as the missing-proof fail-closed
   case.
9. **Exact insertion point:** a block guarded by `if token.pop_secret is not None:`
   placed immediately **after the C2 bound-check failure return** and **before**
   `identity_proven = True`. Every PoP failure reuses the existing
   `_record_rejection(REASON_SUBJECT_TOKEN_INVALID, …)` + `_pre_audit_error(403,
   "forbidden", "subject token is not valid")` — no new reason, no leak. Gate
   strictly on `is not None` (never truthiness).
10. **pop_secret no-leak:** `pop_secret` is written by `insert` and read by
    `get_by_hash` (for verification) only. It must be **excluded** from
    `list_subject_tokens`/`get_by_id` CLI projections (or omitted from the CLI
    output dict). It never appears in any audit row or response except the one-time
    mint 201 body.
11. **At-rest note extension:** a read-only DB compromise still cannot forge enforce
    (the raw token, never stored, is also required); a DB-**write** compromise is
    already game-over for the auth store, so `pop_secret` adds no new write-primitive
    exposure. Long-term: derive per-token secrets from a server master key.
