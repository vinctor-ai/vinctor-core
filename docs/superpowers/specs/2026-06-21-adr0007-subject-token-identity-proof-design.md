# ADR 0007 Identity-Proof — Model 2 (Grant-Bound Subject Tokens)

Date: 2026-06-21
Status: Design approved (founder, 2026-06-21); hardened after a 3-lens adversarial
review (security / API-design / implementation-fit, 2026-06-21). Next:
writing-plans → TDD implementation.

Implements the **Model 2** identity-proof option from
[ADR 0007](../../decisions/0007-delegated-enforce-and-pep-identity.md). The
delegated-enforce mechanism (PEP key, `/v1/enforce/delegated`, separate audit
principal) already ships; today the PEP only *asserts* the subject agent. This
slice lets the subject be *proven* by a Vinctor-issued, grant-bound,
audience-scoped, short-lived token.

## Problem

`/v1/enforce/delegated` authenticates the PEP (via its `pep_` key) and trusts the
PEP's body assertion of who the subject is (`agent_id`, `workspace_id`,
`grant_ref`). Nothing proves the access actually originates from that agent — a
PEP can name any subject in its own workspace. ADR 0007 deferred *how* the subject
is proven; the founder chose Model 2.

## Goal / Non-goals

**Goal:** prove the subject agent on a delegated enforce by requiring a
Vinctor-issued token bound to (subject agent, workspace, one grant, an intended
audience PEP, an expiry). The token upgrades the PEP's assertion from *claimed* to
*proven*, without weakening tenant isolation.

**Non-goals (this slice), each a noted follow-up:**
- mTLS / DPoP proof-of-possession enforcement (leave a forward-compatible
  extension point only).
- Single-use / per-action-bound tokens (reuse within the short TTL is allowed;
  see Risks).
- Explicit token revocation (short TTL + live grant re-evaluation cover it —
  see the "token validity ≠ access" invariant below).
- A "require proven identity" enforcement flag (mirrors ADR 0009's opt-in
  pattern; separate slice).
- JWT / stateless signed tokens; SSO / federation (Model 3).

## Approach: opaque, DB-stored, hashed token

A token is an opaque random string with prefix `vat_` (Vinctor Access Token),
generated by the existing `_new_key` helper (`f"{prefix}{token_urlsafe(24)}"`,
`keys.py`) and stored exactly like the existing key types (per ADR 0006): only the
SHA-256 hash is persisted; the raw value is shown once at mint and is
unrecoverable.

*Alternative considered — signed JWT/PASETO (stateless):* rejected for this slice.
It adds signing-key management and a crypto dependency for no benefit in the
current local/single-process posture; the token can be swapped to a signed format
later without changing the delegated-path contract.

## Token model

A subject token binds, immutably at mint time:

| field | meaning |
|---|---|
| `token_id` | stable **public** id, prefix **`vtk_`** (distinct from the raw `vat_` prefix so the two can never be confused / prefix-collide); safe to log/audit |
| `token_hash` | SHA-256 of the raw `vat_` token; the raw is never stored |
| `workspace_id` | the subject's workspace (= the minting agent's workspace) |
| `agent_id` | the subject agent |
| `grant_ref` | exactly one grant the token authorizes checks against |
| `audience` | the intended PEP's `pep_id` (replay scope; see audience binding) |
| `issued_at`, `expires_at` | validity window |
| `created_by` | minting principal (the agent) |

- **TTL:** default `DEFAULT_SUBJECT_TOKEN_TTL_SECONDS = 300` (named module
  constant, referenced by both the HTTP handler and the CLI default). Operator
  ceiling `VINCTOR_SUBJECT_TOKEN_MAX_TTL_SECONDS` (default 3600). Mint rejects
  `ttl > max` or `ttl <= 0`.
- **Expiry clamp:** `expires_at = min(now + ttl, grant.expires_at)` when the grant
  has an `expires_at`, so a token never outlives its grant.
- **Reuse:** valid for repeated checks within its window (not single-use).
- **Grant-bound:** one `grant_ref`; the PDP still evaluates the actual
  `action`/`resource` against that grant at enforce time.

## Storage (repository pattern + schema version)

Follow the **grant/auto-approval repository pattern**, not the standalone-key
pattern: a `SubjectTokenRepository` Protocol with an `InMemorySubjectTokenRepository`
and a `SQLiteSubjectTokenRepository` twin, wired as a **field on both
`InMemoryV1Service` and `SQLiteV1Service`** (the delegated/mint contracts are
unit-tested through `InMemoryV1Service`, so an in-memory twin is required). The key
repository is the precedent for **hashing / no-raw-persistence only**, not for the
wiring location.

SQLite: add a `subject_tokens` table inside `init_sqlite_schema` via
`CREATE TABLE IF NOT EXISTS` (a whole new table → no per-column migration helper
needed; existing DBs upgrade simply by re-opening). Add a **schema `version 3`**
`INSERT OR IGNORE` row so `schema_versions` reflects the change for
`operator service info` / backup / restore validation.

## Issuance path (agent → Vinctor)

The agent authenticates with its **own** `aak_` key and asks Vinctor to mint a
token for a grant it owns.

- **HTTP:** `POST /v1/tokens`, header `X-Agent-Key`, body
  `{grant_ref, audience, ttl_seconds}`. Success → **HTTP 201** (mirroring
  `POST /v1/grant-requests`), body `{token, token_id, expires_at}`.
- **CLI:** `vinctor agent token mint --grant-ref <ref> --audience <pep_id>
  [--ttl <dur>]`, placed under the existing `agent` role beside `agent enforce`.
  It is an **HTTP client call** to `POST /v1/tokens` with `X-Agent-Key` (mirrors
  the `_agent` HTTP pattern — **not** a direct-SQLite call). Output reuses the
  `operator keys rotate` once-shown convention: a text block with `token_id`,
  `expires_at`, the raw token, and a "store now; cannot be recovered" warning,
  plus a JSON body `{token, token_id, expires_at}`.

**Validation + error taxonomy** (distinct status codes, matching existing
surfaces):
1. `X-Agent-Key` absent/unresolvable → **401** `authentication_required` (mirror
   `v1_http.py`; also call `record_auth_failure(surface="tokens", …)` per ADR 0008).
2. Missing/empty `grant_ref`/`audience`, non-numeric or `ttl_seconds <= 0`, or
   `ttl_seconds > max` → **400** (e.g. `invalid_request` / `ttl_exceeds_max`).
   NB `ttl_seconds` is numeric, so it does **not** go through the string body
   parser and needs its own validation.
3. Grant missing, **not valid**, or not owned by the agent → **403** generic
   `forbidden`, no `grant_ref`/id leak (security-sensitive; stays opaque).

**Grant validity predicate (precise).** Resolve the grant by ref and check it
directly (do **not** use `lookup_grant`, which filters on `workspace_id` only and
omits `agent_id`): the grant must satisfy
`grant.agent_id == agent_id AND grant.workspace_id == workspace_id` (mirror the
hand-rolled equality at `v1_enforce.py:54`) **and be valid**, where *valid* is
defined identically to enforce: `status == 'active' AND status not in {revoked,
expired} AND (expires_at is None OR expires_at > now)`. This matters because the
codebase **never flips `status` to `expired`** — expiry is evaluated lazily only
inside `evaluate_enforce` — so a `status == 'active'` check alone would mint a
fresh proven-identity token against a time-expired grant.

**Mint audit.** A successful mint emits an operator-readable
`subject_token_minted` audit event recording `agent_id`, `workspace_id`,
`grant_ref`, `token_id`, `audience`, `expires_at` — **never the raw token**.
(`grant_ref` in an audit row is fine; it is the agent's own grant and audit rows
are operator-read. The non-disclosure rule below is about cross-tenant responses /
rejection rows.)

## Delegated enforce validation (PEP presents the token)

The token is an **additive optional header** — the existing 5-field request body
`{workspace_id, agent_id, grant_ref, action, resource}` is **unchanged**, so the
path stays backward-compatible and purely additive (ADR 0007's ethos).

PEP calls `/v1/enforce/delegated` with `X-PEP-Key` + the unchanged body **plus the
new optional header `X-Subject-Token`** (and the existing optional
`X-Vinctor-Boundary-Id`).

**When `X-Subject-Token` is present (proven path):**
1. Authenticate the PEP key → `trusted_ws`, `pep_id` (unchanged). Missing/invalid
   `X-PEP-Key` → 401 **before** any token logic, even if a token is present.
2. Resolve the token by hash. **Token not found OR the token store raises an
   error → fail CLOSED** (403 `forbidden` + rejection audit) — never 503, never
   fall back to the legacy asserted path. (The existing enforce paths turn a repo
   exception into 503; the proof gate must not, or an error degrades a proven
   request to unproven/open.)
3. Token not expired.
4. **Audience match:** `token.audience == pep_id` **and** `token.workspace_id ==
   trusted_ws` — the (workspace, pep_id) pair must match the authenticated PEP, so
   a `pep_id` reused/colliding across workspaces cannot match.
5. **Single identity tuple (the core invariant).** Resolve the grant by
   `grant_ref` and require ALL of these to be equal — any inequality fails closed:
   - `token.workspace_id == trusted_ws == body.workspace_id == grant.workspace_id`
   - `token.agent_id == body.agent_id == grant.agent_id`
   - `token.grant_ref == body.grant_ref == grant.grant_ref`

   Grant ownership is checked against this one resolved tuple — never mixing
   token-vs-body sources in different places.
6. Run the PDP `enforce(action, resource)` on the live grant (re-fetched via
   `grant_repository.get_by_ref` + `evaluate_enforce`).
7. The decision is recorded **identity-proven** in the audit trail.

Any failure in steps 2–5 fails closed with the existing generic `forbidden`
response (no `grant_ref`/id/token leak) and an operator-only rejection audit event
(`reason_code = subject_token_invalid`), per ADR 0008.

**When `X-Subject-Token` is absent (legacy path):** behavior is exactly as today —
the subject is asserted, the decision is **not** identity-proven. No new claim, no
broken caller. (Documented: proven on-behalf-of identity requires the token.)

**Invariant — token validity ≠ access.** A valid subject token proves *who* is
asking, not *whether* access is granted: the live grant is always re-evaluated by
the PDP at enforce time, so a revoked/expired grant denies even with a
still-valid token. This is what makes the "no explicit revocation" non-goal safe;
it must be tested, not assumed.

## Audit & redaction invariants

- Add two **dedicated additive fields** to `AuditEvent`: `subject_token_verified: bool =
  False` and `token_id: str | None = None`. Emit them in `to_dict()` only when set
  (mirroring `enforcing_principal`), and read them back in
  `_audit_event_from_json`. Add the corresponding params to `build_audit_event` /
  `AuditEventInput`. **Do not overload `reason_code`** — it is reserved for ADR
  0008 rejection events; overloading it would make it mean two things by
  event_type. No new SQLite column is required (the fields live inside
  `event_json`) unless we later want `operator audit list` filtering by proven —
  out of scope here.
- Token-validation failures emit an operator-only rejection event
  (`reason_code = subject_token_invalid`); caller response stays generic.
- **Never logged or returned in any response / rejection row:** the raw token, the
  raw key, internal ids, tool input. The raw token is shown once at mint only.
  (`grant_ref` non-disclosure applies to cross-tenant/PEP responses and rejection
  rows — `build_rejection_audit_event` zeroes it — **not** to the agent's own mint
  request or the `subject_token_minted` decision-style audit row.)

## Configuration

- `DEFAULT_SUBJECT_TOKEN_TTL_SECONDS = 300` — named module constant; default mint
  TTL when `--ttl`/`ttl_seconds` omitted.
- `VINCTOR_SUBJECT_TOKEN_MAX_TTL_SECONDS` (default 3600) — mint TTL ceiling
  (matches the `VINCTOR_*` convention in `service_config`).

## Implementation wiring checklist

Neither `/v1/tokens` nor `X-Subject-Token` is reachable today; enumerate the
touchpoints so the plan does not miss one:

- **Router:** new exact-path branch for `POST /v1/tokens` in
  `local_http.py:_handle_request` + a `handle_v1_tokens_http` handler + a
  `V1TokenService` Protocol (uses the existing agent-identity resolver).
- **Delegated proven path:** add an optional `subject_token` to
  `V1DelegatedEnforceRequest`; read `X-Subject-Token` in
  `handle_v1_delegated_enforce_http`; thread it through
  `delegated_enforce_v1_contract`.
- **Services:** add `subject_token_repository` + a `mint_subject_token` method to
  both `InMemoryV1Service` and `SQLiteV1Service`; the delegated method consults the
  repo when a token is supplied.
- **Runtime resolvers:** the proven path needs the PEP resolver. `service serve`
  wires it (`service_runtime.py`); **`local start` does not wire the PEP resolver
  today** (`local_launcher.py`) — wire the PEP resolver (and the token repo) into
  **both** runtimes so mint *and* the proven delegated path are reachable via
  `local start` and `service serve`. (Mint itself only needs the agent resolver,
  which both runtimes already wire.)

## Testing (TDD)

**Mint:**
- mint for an owned, **valid** grant → `vat_` token + `vtk_` id + clamped expiry;
  raw shown once; `subject_token_minted` audit row (no raw token).
- mint for another agent's grant → 403, no token created.
- mint for a **revoked** grant → 403 (distinct from missing).
- mint for a **time-expired but status=='active'** grant → 403 (the status-vs-valid
  gap is explicitly covered).
- mint with `ttl > max`, `ttl <= 0`, or non-numeric ttl → 400.
- mint with missing/empty `grant_ref` or `audience` → 400.
- bad/absent `X-Agent-Key` → 401 + `auth_failed` (surface=tokens).

**Delegated + token:**
- valid token, matching audience + identity tuple, in-bounds action → permit;
  audit `subject_token_verified=true`, `token_id` recorded; raw token absent everywhere.
- expired token → fail closed `forbidden` + rejection audit `subject_token_invalid`.
- token store raises during lookup → fail closed `forbidden` (NOT 503, NOT legacy
  fallback).
- audience mismatch (token for PEP A presented at PEP B) → fail closed.
- token workspace ≠ trusted PEP workspace (cross-workspace) → fail closed, no
  audit leak.
- token subject ≠ asserted body subject → fail closed.
- token+body agree on agent but the **grant is owned by a different agent** in the
  same workspace → fail closed.
- `X-PEP-Key` missing while `X-Subject-Token` present → 401 before token logic.
- token present but a required body field missing → existing 400.
- **grant revoked after mint, before enforce → enforce denies** (proves the "token
  validity ≠ access" invariant / safe deferred-revocation non-goal).
- **no token → legacy asserted path unchanged** (regression guard,
  `subject_token_verified` false/absent).
- round-trip: `subject_token_verified` / `token_id` survive SQLite persistence (the
  `enforcing_principal` round-trip is the precedent).

## Risks / open notes

- **Audience discovery:** the agent must know the target `pep_id` to mint. In the
  local/preview posture the operator provisions the PEP
  (`operator keys rotate pep`) and shares its id via the agent's config.
  `pep_id` is a **stable public identifier**, not a secret: audience binding
  provides *scope limitation*, not secrecy. Matching on the (workspace, pep_id)
  pair prevents cross-workspace collisions; the real anti-replay control is
  PoP/mTLS (deferred).
- **Replay within TTL — honest blast radius:** the token is grant-bound but **not**
  action/resource-bound, so within its (short, clamped) TTL a presented token is a
  bearer credential for the grant's **entire scope** at the bound PEP — not a
  single operation. That is why the TTL is short and PoP/mTLS is the planned
  hardening; per-action token binding is a possible future extension point.
